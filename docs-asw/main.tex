\section{Introduzione}
Il progetto HiHome consiste in un applicativo per gestire dispositivi all'interno di una smart home. Permette di gestire dispositivi smart da applicazione e aggiungere automazioni. Sono presenti funzionalità come notifiche, log di event relativi ai device smart, informazioni sullo stato attuale di temperatura, meteo e qualità dell'aria nella zona e grafici relativi ai valori passati di questi ultimi. Inoltre, è integrato con un agente AI in modo da effettuare alcune azioni in linguaggio naturale. L'applicazione supporta due tipi di utenti:

\begin{itemize}
    \item \textbf{Admin}: ha accesso a funzioni avanzate sulla gestione della casa, come log di eventi relativi a manipolazione di smart device, creazione di automazioni e aggiunta di device.
    \item \textbf{Utente Standard}: può agire sugli smart device, visualizzare statistiche relative allo stato attuale della casa, grafici relativi ad eventi passati, come lo stato della qualità dell'aria nella zona durante la settimana precedente e interagire con il chatbot.
\end{itemize}

\subsection{Design Architetturale}
Il progetto è stato realizzato seguendo la filosofia di design \textbf{Domain-Driven Design (DDD)}. Come pattern architetturale, è stata adottata un'architettura esagonale (ports and adapters) per ogni bounded context identificato. In particolare, sono stati identificati i seguenti bounded contexts:
\begin{itemize}
    \item \textbf{Home context}: è il contesto principale dell'applicazione, che si occupa della gestione delle case, dei dispositivi smart e della chat
    \item \textbf{User context}: si occupa della gestione degli utenti e dell'autenticazione
    \item \textbf{Notification context}: si occupa della gestione delle notifiche
    \item \textbf{Rule context}: si occupa della gestione delle regole di automazione
    \item \textbf{Weather context}: si occupa di ottenere dati relativi al meteo e alla qualità dell'aria da servizi esterni
\end{itemize}
L'architettura utilizza lo stack MEVN, con i primi 4 contesti implementati nel backend in Node.js, mentre il Weather context è implementato in un servizio esterno in Go.

L'applicazione HiHome è quindi divisa in diversi componenti. I client, che accedono all'applicazione tramite interfaccia web, il backend, che espone API RESTful per la comunicazione con i client e gestisce la business logic, e il servizio esterno per il meteo, che è interrogato periodicamente dal backend per ottenere dati aggiornati relativi alle condizioni nelle zone in cui le varie case si trovano. Il server backend, inoltre, accetta connessioni WebSocket dai client per l'invio di notifiche e aggiornamenti in tempo reale, e per lo streaming delle risposte del modello AI. Il backend accede ai dati tramite un database MongoDB.

Per quanto riguarda i concetti di dominio, si definiscono di seguito i principali, già brevemente descritti in precedenza:
\begin{itemize}
    \item \textbf{Casa}: rappresenta una casa intelligente, con i suoi dispositivi smart, le sue automazioni e i suoi utenti
    \item \textbf{Dispositivo Smart}: rappresenta un dispositivo intelligente all'interno di una casa, come una luce, una finestra o un termostato
    \item \textbf{Automazione}: rappresenta una regola di automazione, composta da una condizione di attivazione e da una o più azioni da eseguire quando la condizione è verificata
    \item \textbf{Utente}: rappresenta un utente dell'applicazione, con i suoi permessi e le sue preferenze
\end{itemize}

\subsection{Logica di Autorizzazione}
Questo middleware rappresenta un componente fondamentale per la sicurezza delle API esposte. Tramite questo codice, il backend si assicura che solo gli utenti con privilegi di "Admin" possano eseguire determinate operazioni, come la manipolazione di regole e l'aggiunta di dispositivi. Il token JWT viene estratto dall'header della richiesta HTTP, decodificato e verificato, permettendo al controller di procedere solo se l'utente risulta avere il ruolo di amministratore.
\begin{lstlisting}
export const adminMiddleware = (
  req: Request,
  res: Response,
  next: NextFunction,
) => {
  const token = getTokenFromHeader(req);
  if (!token) {
    res.status(401).json({ error: "Unauthorized: Missing or invalid token" });
    return;
  }

  try {
    const decoded = verifyAuthToken(token) as JwtPayload;
    (req as any).user = decoded;
    if (decoded.role !== "Admin") {
      res.status(403).json({ error: "Forbidden: Admin access required" });
      return;
    }
    next();
  } catch (error) {
    res.status(401).json({ error: "Unauthorized: Invalid token" });
  }
};
\end{lstlisting}

\subsection{Validazione Input tramite Middleware}
Questo blocco di codice illustra l'utilizzo della libreria \texttt{express-validator} per convalidare i parametri in ingresso alle API prima che vengano elaborati dai controller. Attraverso una catena di controlli dichiarativi, il middleware garantisce l'integrità dei dati: ad esempio, verifica che il parametro \texttt{temperature} sia un numero compreso tra 5 e 40 e che venga fornito solo se l'azione richiesta è effettivamente \texttt{setTemperature}. Questo approccio previene errori logici, iniezioni di dati malformati e disaccoppia la validazione sintattica dalla business logic.
\begin{lstlisting}
export const temperatureValidator = [
  body("temperature")
    .if((_value, { req }) => req.params?.action === "setTemperature")
    .notEmpty()
    .withMessage("temperature is required when action is setTemperature")
    .isNumeric()
    .withMessage("temperature must be a number")
    .isFloat({ min: 5, max: 40 })
    .withMessage("temperature must be between 5 and 40"),

  body("temperature")
    .if((_value, { req }) => req.params?.action !== "setTemperature")
    .isEmpty()
    .withMessage(
      'temperature should only be provided when action is "setTemperature"',
    ),

  validate,
];
\end{lstlisting}

\subsection{Notifiche Real-Time con Socket.io}
Questo breve snippet mostra l'adattatore utilizzato per inviare aggiornamenti di stato dei dispositivi in tempo reale ai client connessi. Quando lo stato di un sensore viene aggiornato nel backend, questo adapter utilizza Socket.io per emetter un evento specifico a tutti i client connessi della casa interessata, garantendo così che le interfacce utente riflettano immediatamente i cambiamenti senza necessità di refresh manuali. Il metodo \texttt{broadcast} centralizza la logica di invio dei messaggi, semplificando l'implementazione di notifiche per i vari tipi di eventi.
\begin{lstlisting}
export class SocketIOSensorUpdateAdapter implements SensorUpdatePort {
  constructor(
    private io: Server,
    private notificationPort?: HomeNotificationOutboundPort,
  ) {}

  private broadcast(homeId: string, event: string, payload: object): void {
    console.log(
      `SocketIOSensorUpdateAdapter: Broadcasting ${event} to home-${homeId}`,
    );
    this.io.to(`home-${homeId}`).emit(event, payload);
  }

  sendInternalTemperatureUpdate(home: Home, update: TemperatureState): void {
    this.broadcast(home.id, "sensor:internal-temperature", {
      temperature: update.temperature,
    });
  }
}
\end{lstlisting}

\subsection{Integrazione con Modello AI (Streaming)}
Questo snippet mostra come viene gestita la comunicazione con il modello AI in tempo reale. Invece di attendere la generazione completa della risposta, il metodo inoltra progressivamente i frammenti di testo generati al client tramite una porta dedicata (\texttt{ChatStreamPort}, che utilizza Socket.io). Prima di interrogare il modello, il contesto viene costruito dinamicamente generando un system prompt che include le direttive specifiche per il modello AI.
\begin{lstlisting}
async streamChat(
  homeId: string,
  username: string,
  userMessage: string,
  history: ChatMessage[],
  streamPort: ChatStreamPort,
  isAdmin: boolean,
): Promise<string> {
  if (!userMessage.trim()) {
    throw new Error("Message is required");
  }

  const messages = await this.buildMessages(homeId, userMessage, history);

  return this.chatCompletionPort.streamChat(
    messages,
    this.options.model,
    homeId,
    streamPort,
    isAdmin,
  );
}

\subsection{Euristiche di Nielsen}
Per quanto riguarda la produzione di mockup e l'implementazione del frontend, è stato prestata particolare attenzione alle \textbf{Euristiche di Nielsen} per l'usabilità, al fine di garantire un'esperienza utente ottimale. Di seguito sono riportati alcuni esempi di come queste euristiche sono state applicate durante la progettazione dell'interfaccia utente:
\begin{itemize}
    \item \textbf{Visibilità dello stato del sistema}: sono state implementate notifiche in tempo reale per informare gli utenti su eventi importanti. Sono stati inseriti feedback visivi immediati quando lo stato di un dispositivo smart viene modificato, ad esempio agendo sul toogle corrispondente.
    \item \textbf{Corrispondenza tra il sistema e il mondo reale}: sono stati utilizzati termini e icone familiari agli utenti per rappresentare i dispositivi smart e le azioni disponibili, ad esempio un'icona di una lampadina per rappresentare una luce.
    \item \textbf{Controllo e libertà dell'utente}: è stata implementata la possibilità di annullare azioni semplicemente, come il cambio istantaneo del toggle di dispositivi smart o la possibiltà di eliminare facilemente regole di automazione. Inoltre, la navbar permette una navigazione semplice e veloce tra le diverse sezioni dell'applicazione.
    \item \textbf{Coerenza e standard}: è stata mantenuta una coerenza visiva e funzionale in tutte le pagine dell'applicazione, utilizzando la stessa palette colori, lo stesso stile dei pulsanti e lo stesso layout per elementi simili.
    \item \textbf{Prevenzione dell'errore}: abbiamo implementato meccanismi di prevenzione di errori, ad esempio, nel momento in cui l'utente agisce sul termostato, la temperatura viene modificata attraverso pulsanti di incremento/decremento, che sono disabilitati quando viene raggiunta la temperatura massima o minima consentita, evitando così l'inserimento di valori non validi.
    \item \textbf{Riconoscimento piuttosto che ricordo}: sono stati utilizzati menu a tendina, icone e pulsanti chiaramente etichettati per facilitare la navigazione e l'interazione, riducendo la necessità per gli utenti di ricordare comandi o percorsi specifici. Ad esempio, nella creazione di una regola di automazione, i campi per la condizione e l'azione sono presentati come menu a tendina con opzioni predefinite, facilitando la selezione e riducendo la possibilità di errori, come l'inserimento di device non esistenti o azioni non valide.
    \item \textbf{Flessibilità e efficienza d'uso}: l'applicazione è stata progettata per essere utilizzata sia da utenti esperti che da utenti meno esperti. Ad esempio, una regola di automazione può essere creata tramite un'interfaccia guidata per utenti meno esperti, ma anche attraverso linguaggio naturale via chat, offrendo così un trade-off tra flessibilità e intuitività.
    \item \textbf{Estetica e design minimalista}: seguendo una filosofia "less is more", l'interfaccia è stata progettata per essere pulita, evitando elementi pesanti e ricchi di testo a favore di icone intuitive all'interno di layout semplici.
    \item \textbf{Aiutare gli utenti a riconoscere, diagnosticare e recuperare gli errori}: sono stati implementati messaggi di errore chiari e specifici, ad esempio, se un utente tenta di eseguire un'azione non consentita, viene visualizzato un messaggio che spiega il motivo del blocco e suggerisce come risolvere il problema.
    \item \textbf{Aiuto e documentazione}: dando la priorità a un design intuitivo ed a un'interfaccia auto-esplicativa, abbiamo ritenuto che l'applicazione non necessitasse di una sezione di aiuto dedicata.
\end{itemize}
\subsection{Unit Testing e Strumenti di sviluppo}
Per garantire il corretto funzionamento delle funzionalità principali, sono stati utilizzati test unitari per il backend. Questo permette di verificare frequentemente lo stato dell'applicazione e di avere sempre a disposizione test regressivi da effettuare ad ogni cambiamento o estensione delle funzionalità. Per i test unitari, sono stati utilizzati \textbf{Supertest} e \textbf{Vitest}. I test sono stati inseriti nella pipeline di Continuous Integration e nei Git Hook, perciò sono eseguiti automaticamente ad ogni push, ad ogni pull request e ad ogni commit sul branch principale, garantendo così che il codice presente in questi branch sia sempre coperto da test e che eventuali errori vengano individuati tempestivamente. Inoltre, i test ci hanno anche permesso di tenere sotto controllo la copertura del codice.
Per assicurarsi la compatibilità cross-browser, l'applicazione è stata testata regolarmente su Google Chrome e Mozilla Firefox. Grazie alla possibilità di cambiare il tipo di device simulato all'interno degli strumenti di sviluppo, è stato possibile testare l'applicazione su una vasta gamma di dispositivi, dando la priorità ai dispositivi mobile, in ottima mobile-first.
